Gestión de Comunicación del Software


¿Qué es? Objetivos Actividades

¿Qué es la gestión de configuración del software?

La gestión de configuración de software es el proceso que identifica, monitorea y almacena todos los artefactos de un proyecto de software, las cuales probablemente cambien y estableciendo relaciones entre dichos cambios.

Estos artefactos son un producto que se va generando durante el proceso de desarrollo de software, estos artefactos se pueden mantienen durante el control de configuración y durante el proces se seleccionaran cuales se van a mantener y cuales no son necesarios. Estos artefactos en la gestión de desarrollo son llamados items de configuración.

Objetivos de la configuració del software

La gestión de configuración de software tiene 4 objetivos importantes los cuales son:


  1. Integridad de la versión actual
  2. Integridad de sobre-escritura
  3. Deshacer
  4. Recuperación de desastres


Integridad de la versión actual

Aceptar cambios en un item de configuración para incorporarlos en la versión actual del software.
Estos cambios se almacenan en un repositorio para que se puedan llegar a recuperar.

Integridad de sobre-escritura

Cada miembro del equipo de desarrollo puede trabajar en simultáneo con los demás en los items de configuración sin que ninguno de estos cambios afecte algo que este realizando algún otro miembro.

Deshacer

Cuando algún item de configuración llega a un estado en el cual posee errores por motivos de algún cambio indebido, se es necesario regresar a una versión previa del proyecto.

Recuperación de desastres

Es un proceso mucho más grande al de deshacer ya que implica guardar versiones previas del software y si es necesario volver a estas versiones.

Actividades en el plan de gestión de configuración de Software

Identificación de la configuración:

Primero se deben de identificar los items de configuración donde cualquier artefacto que va a ser modificado lo puede ser, durante todo el proceso se toma la información que se asocia a dicho item como lo es la fecha de modificación, las versiones, los autores y el estado de cada item.
Cuando un artefacto pasa por varias modificaciones hasta llegar a ser estable y que pueda usarse para el siguiente paso de desarrollo dandole el nombre de Baseline. Cuando se tiene un conjunto de Baseline se conforma un hito de proyecto lo cual significa que se puede avanzar en el número de versiones del software.


Número de versiones del software:

Una de las formas más usadas y la que más se recomienda es la siguiente:

Major.Minor.Patch


Donde "Patch version" sólo cambia cuando se realizan modificaciones al software y soluciona algunos errores que se esten presentando.
"Minor Version" sólo cambia cuando se realizan modificaciones al API que agregan funcionalidades nuevas que mantienen compatibilidad con las anteriores.
Y "Major version" sólo cambia cuando se realizan modificaciones Al API que agurefan funcionalidades nuevas pero que no mantienen compatibilidad con las anteriores.

Se usan algunos múmeros para las versiones del software.

  • 0 se usa para versiones alfa.
  • 1 se usa para versiones beta.
  • 2 se usa para versiones candidatasa.
  • 3 se usa para versiones en producción.

Ejemplos:

  • 1.3.0.3 => Versión alfa del patch 3.
  • 1.3.1.3 => Versión beta del patch 3.
  • 1.3.2.3 => Versión cadidata a lanzamiento del patch 3.
  • 1.3.3.3 => Versión release del patch 3.
